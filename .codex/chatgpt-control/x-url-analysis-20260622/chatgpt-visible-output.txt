重要度順の結論

本番エージェントの勝ち筋は「モデル」より「基盤設計」
token efficiency、state-aware RAG、memory、skill lifecycle、harness分割が共通テーマ。モデルを強くするだけでは、コスト・ノイズ・局所最適・長期運用問題は解決しない。

RAGは「検索を増やす」から「ノイズを減らし、状態を管理する」へ移っている
GoogleのAgentic RAG、AmazonのState-Aware RAG、LLM-Oriented IRは、どれも「十分な根拠」「working memory」「検証可能性」を重視している。

実装/運用に使えそうなアイデア

Codex用の標準フロー
grill-me → plan.md → 複数観点レビュー → 実装 → test → 差分レビュー → memory/skill更新 に分ける。
plan承認を一括にせず、未決事項を解消してから実装へ進める。

ハーネスは小さく分ける
research-harness、coding-harness、review-harness、rag-harness、release-harness のように責務分離する。万能ハーネスは避ける。

RAGにworking memoryを入れる
検索結果をそのまま積むのではなく、候補 → 関連性判定 → working memory更新 → sufficient context判定 → 回答 にする。

永続メモリは検索型にする
全記憶を常時コンテキストに入れず、ローカルMarkdown/qmd/ベクトル検索などで必要分だけ注入する。各記憶には作成日、根拠、信頼度、失効条件を持たせる。

Skillはテスト可能な資産にする
MUSE-Autoskill的に、各Skillに README, examples, unit tests, failure cases, revision log を持たせる。

出力はDSL/構造化中間表現を使う
図、WBS、レビュー表、要件定義は、AIに直接整形させるのではなく、YAML/JSON/DSLへ落としてvalidate/renderする。

過剰実装抑制ルールを入れる
Ponytail的に、実装前に「既存機能・stdlib・native feature・既存依存で済むか」を確認させる。ただし安全性、データ破壊防止、アクセシビリティは削減対象外にする。

採用しない方がよいもの

単一巨大ハーネス
短期的には速いが、密結合・検証困難・修正不能になりやすい。

AIだけにDesign/Architecture判断を丸投げ
Format/Lint/Styleは任せやすいが、Architectureやドメイン判断は人間の責任境界を残すべき。
Zenn

検索結果を無制限に積むRAG
ノイズが増え、ハルシネーションや推論失敗を招く。denoisingとworking memoryが必要。

「観察マスキング」を一律適用
モデル/検索器/タスクの組み合わせで逆効果になり得る。評価なし採用は危険。
Hugging Face

「無限メモリ」という売り文句の鵜呑み
実態は検索可能な外部記憶。古い記憶、誤記憶、不要記憶、権限付き情報の管理が必要。

AIに図や資料を直接描かせる運用
再現性が低い。DSL/validate/renderの流れに寄せる方が安定する。
